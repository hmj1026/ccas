## 背景 (Context)

在 `parser-engine` 之後，系統會擁有結構化交易資料，但 `Transaction.category` 尚未有穩定來源。產品規格明確要求採用 rule-based keyword classification，並透過 `categories` 資料表管理商家關鍵字與分類名稱。這個 change 需要把分類規則定義清楚，避免未來 API 與 dashboard 依賴一套不穩定或不可重現的分類邏輯。

## 目標 / 非目標 (Goals / Non-Goals)

**目標：**
- 以 `categories` 資料表作為分類規則唯一來源
- 定義可預期、可重現的關鍵字匹配規則
- 為無匹配結果的交易提供固定的預設分類
- 將分類結果套用到新交易與既有交易

**非目標：**
- 不使用機器學習或外部分類服務
- 不在此 change 中建立分類管理 API / UI
- 不處理多層分類樹或使用者自訂優先權欄位

## 決策 (Decisions)

### D1: `categories` 資料表是分類規則唯一來源

**選擇**: 所有分類規則都從 `categories` 資料表讀取，不使用 YAML 或硬編碼規則。

**理由**: 專案已經定義 `categories` 表，未來 Settings 頁面也需要管理同一份資料，應避免雙重來源。

**考慮過的替代方案**:
- YAML 檔：與資料庫狀態不同步
- 程式內硬編碼：維護成本高

### D2: 匹配採「最長關鍵字優先」，同長度時以較小 `id` 決定

**選擇**: 若同一商家名稱匹配到多個關鍵字，優先採用最長關鍵字；若長度相同，採用 `categories.id` 較小者。

**理由**: 目前 schema 沒有 priority 欄位，需要一個 deterministic 規則。最長關鍵字通常語意更精確。

**考慮過的替代方案**:
- 單純第一筆命中：順序不明確
- 另加 priority 欄位：目前不是必要前提

### D3: 字串匹配先做正規化

**選擇**: 比對前先做去除前後空白、壓縮重複空白與 ASCII 大小寫不敏感處理。

**理由**: 可降低銀行帳單與商家名稱格式差異對匹配結果的影響。

**考慮過的替代方案**:
- 完全不正規化：命中率較差
- 做過度清洗：可能誤傷有效資訊

### D4: 無命中時使用固定分類 `未分類`

**選擇**: 沒有任何規則命中的交易，一律標記為 `未分類`。

**理由**: 這比 null 更容易在報表、Telegram 與設定頁面中辨識與追蹤。

**考慮過的替代方案**:
- 保留 null：前端與查詢處理較複雜
- 直接丟棄交易：不可接受

### D5: 分類引擎支援初次套用與重跑

**選擇**: 相同分類邏輯可用於新交易寫入時，也可用於既有交易的 reclassification。

**理由**: 分類關鍵字會持續調整，若無法重跑，歷史資料會失去一致性。

**考慮過的替代方案**:
- 僅對新交易分類：舊資料無法追補
- 每次查詢時即時計算：成本高且不穩定

## 風險 / 取捨 (Risks / Trade-offs)

**關鍵字衝突** → 用最長關鍵字與固定 tie-break 規則保證 deterministic 行為  
**規則覆蓋不足** → 以 `未分類` 顯示缺口，方便後續補規則  
**分類重跑可能改變歷史結果** → 規格接受這是規則更新帶來的預期行為
