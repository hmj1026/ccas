## 背景 (Context)

在 gmail-ingestor、parser-engine、keyword-classifier 與 telegram-bot 各 change 完成後，每個功能模組都有各自定義好的 job 入口與輸出格式，但尚未有任何元件負責把它們串接成一個完整的端到端流程。使用者目前無法以單一指令執行完整的帳單處理週期，也沒有機制在帳單到期前自動發出提醒。

此 change 的核心職責是作為最終協調者，定義各階段的呼叫順序與資料傳遞方式，建立穩定的排程機制，並提供手動觸發介面。因為此 change 依賴所有其他功能 change，因此應在所有其他 change 完成後最後實作。

## 目標 / 非目標 (Goals / Non-Goals)

**目標：**
- 定義五階段 pipeline 的執行順序：ingest → decrypt → parse → classify → notify
- 確保各階段部分失敗不阻斷整體 pipeline，已成功項目繼續流入下一階段
- 整合 APScheduler，提供可設定頻率的週期性自動觸發
- 建立付款到期提醒排程工作（到期前 3 天與 1 天）
- 提供 CLI 指令與 REST API 端點進行手動觸發
- 回傳包含各階段統計與總耗時的結構化 pipeline 摘要

**非目標：**
- 不修改任何單一階段的內部邏輯（ingest、decrypt、parse、classify、notify 各自負責）
- 不提供 pipeline 執行歷史的前端頁面（屬於 frontend-dashboard change 範疇）
- 不實作分散式工作佇列或非同步任務系統
- 不提供各階段的個別 API 端點

## 決策 (Decisions)

### D1: 同步循序協調器

**選擇**: `run_pipeline()` 採同步循序方式依序呼叫每個階段，每個階段完成後才進入下一個。

**理由**: 五個階段存在明確的資料依賴關係（每個階段的輸入來自前一階段的輸出），平行化反而會增加複雜度與錯誤情境。對個人工具的帳單處理量而言，同步執行效能已完全足夠。

**考慮過的替代方案**:
- 異步工作佇列（Celery / RQ）：對帳單量過於複雜，難以偵錯
- 各階段平行觸發：不適用，因為資料依賴鏈要求嚴格順序

### D2: 五階段 pipeline：ingest → decrypt → parse → classify → notify

**選擇**: pipeline 定義為五個有序階段，其中 decrypt 為獨立的第二階段，處理密碼保護 PDF 的解密。

**理由**: 把解密從 ingest 與 parse 中抽出是必要的，因為解密需要銀行專屬密碼，且部分 PDF 可能無需解密。獨立為一個階段可讓 parser 只需處理已解密的明文 PDF，分離各自關切面。

**考慮過的替代方案**:
- 在 ingest 階段直接解密：ingest 應只負責下載，不應知道密碼格式
- 在 parse 階段嘗試解密：parse 邏輯更複雜，錯誤責任不清楚

### D3: 各階段獨立容錯，部分失敗不阻斷後續階段

**選擇**: 每個階段以列表為單位處理輸入項目，每筆項目各自 catch 錯誤；失敗項目被記錄但不再傳遞給下一階段，已成功的項目繼續流入下一階段。

**理由**: 對個人帳單工具而言，一張帳單解析失敗不應導致其他銀行帳單的完整流程停止。這種容錯模式延續了 gmail-ingestor 中已建立的批次容錯設計原則。

**考慮過的替代方案**:
- Fail-fast：任何一筆失敗就中止整個 pipeline：對多銀行環境實用性差
- 完全吞掉所有錯誤：不利觀測與後續補救

### D4: APScheduler 週期性觸發，設定以 Settings 為來源

**選擇**: 使用 APScheduler 的 `BackgroundScheduler` 建立週期性 pipeline 排程工作。觸發頻率（cron 表達式或 interval）由 `Settings` 統一管理，而非硬編碼。

**理由**: APScheduler 是 Python 生態系中成熟、輕量的排程函式庫，不需要額外的外部基礎設施（對比 Celery 需要 Redis/RabbitMQ）。設定外部化可讓使用者調整排程頻率而無需修改程式碼。

**考慮過的替代方案**:
- 系統 cron job：不跨平台，部署更複雜
- Celery Beat：對個人工具過重，需要 broker
- 硬編碼排程頻率：彈性差，修改需要重新部署

### D5: 付款提醒為獨立排程工作，不屬於 pipeline 流程

**選擇**: 付款到期前 3 天與 1 天的提醒通知以獨立的 APScheduler 工作實作，每日固定時間執行，查詢 `due_date` 落在目標日期的未付帳單並觸發 Telegram 通知。

**理由**: 付款提醒的觸發條件（帳單到期日）與 pipeline 的觸發條件（有新郵件可處理）無關。混入 pipeline 流程會讓提醒邏輯與帳單資料流混淆，獨立工作更易測試與維護。

**考慮過的替代方案**:
- 在 pipeline 最後一個階段附加提醒邏輯：職責不清，提醒會受 pipeline 是否有新資料影響
- 前端定時輪詢到期帳單：需要前端保持運作，不適合背景自動化

### D6: CLI 模組入口與 API 端點雙管道手動觸發

**選擇**: 提供兩種手動觸發方式：CLI（`uv run python -m ccas.pipeline`）與 API 端點（`POST /api/pipeline/trigger`，需 Bearer token 驗證），兩者均回傳相同的 pipeline 摘要結構。

**理由**: CLI 適合開發期偵錯與手動補跑，API 端點則供前端 dashboard 的「立即執行」按鈕使用。共用同一個 `run_pipeline()` 實作可確保兩個觸發路徑的行為完全一致。

**考慮過的替代方案**:
- 只提供 CLI：前端無法觸發，使用者體驗差
- 只提供 API 端點：開發期偵錯不便

## 風險 / 取捨 (Risks / Trade-offs)

**此 change 依賴所有其他 change** → 應最後實作；各階段介面需依賴其他 change 定義好的 job 入口，若介面未穩定則需調整
**APScheduler 在程序重啟後失去工作狀態** → 每次程序啟動時重新註冊所有排程工作，排程設定由 Settings 持久化確保一致性
**同步執行可能在 PDF 量大時阻塞** → 個人工具每月帳單量有限，同步執行效能足夠；未來有需求再考慮異步化
**各階段部分失敗可能導致資料不一致** → 各階段均保存處理狀態到資料庫，可追蹤哪些項目在哪個階段失敗，並支援重跑
**付款提醒可能重複發送** → 提醒工作應在觸發前檢查是否已發送過（基於帳單 id 與提醒類型的唯一鍵），避免重複通知
