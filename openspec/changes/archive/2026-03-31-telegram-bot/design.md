## 背景 (Context)

產品規格將 Telegram Bot 定位為優先互動介面，必須支援帳單狀態查詢、到期提醒、標記已繳與消費摘要。這些能力都依賴解析與分類後的資料，但 bot 本身仍需要獨立定義 command surface、文字回覆格式與主動通知事件，避免後續 scheduler 或 API change 把責任混在一起。

## 目標 / 非目標 (Goals / Non-Goals)

**目標：**
- 提供 `/status [all|unpaid|paid]`、`/upcoming`、`/paid {bill_id}`、`/summary {YYYY-MM}`、`/category {YYYY-MM}`
- 定義新帳單解析完成、到期前提醒與解析失敗通知
- 讓 `/paid` 能安全更新帳單狀態
- 讓所有 bot 回覆都保持簡潔、可讀、適合行動裝置

**非目標：**
- 不在此 change 中定義排程如何觸發通知
- 不提供非 Telegram 的互動通道
- 不處理複雜對話流程、多步驟 wizard 或 inline keyboard

## 決策 (Decisions)

### D1: 指令面固定對齊產品規格

**選擇**: v1 只支援 `/status`、`/upcoming`、`/paid`、`/summary`、`/category` 這 5 個指令。

**理由**: 這組指令已直接對應產品需求，足以覆蓋本月帳單狀態、近期到期、已繳標記與摘要查詢。

**考慮過的替代方案**:
- 提前加入更多管理指令：會擴大 scope
- 只做通知不做查詢：無法滿足使用者即時查詢需求

### D2: 回覆格式採純文字摘要優先

**選擇**: 所有指令回覆與通知都以簡潔文字清單為主，不依賴富互動元件。

**理由**: Telegram 純文字訊息最穩定，也更容易快速瀏覽與轉傳。

**考慮過的替代方案**:
- Inline keyboard：互動性較高，但初期維護成本較大
- 長格式報表：在手機上不易閱讀

### D3: `/status` 與 `/upcoming` 使用固定時間語意

**選擇**: `/status` 預設查詢當月帳單，`filter` 預設為 `all`；`/upcoming` 固定查詢未來 7 天內未繳帳單。

**理由**: 這和產品規格一致，也能避免指令參數設計變得過度複雜。

**考慮過的替代方案**:
- 讓所有查詢都要帶日期參數：使用負擔較高
- 把 `/upcoming` 做成可配置天數：目前非必要

### D4: `/paid` 是冪等的單向標記操作

**選擇**: `/paid {bill_id}` 只負責將帳單標記為已繳；若帳單已是已繳，回覆已處理狀態，而不是報錯。

**理由**: Bot 操作應該盡量簡單且容錯。冪等行為能避免使用者重複操作造成困惑。

**考慮過的替代方案**:
- 支援 toggle：指令語意變模糊
- 重複標記視為錯誤：使用體驗較差

### D5: 通知內容由獨立 rendering 層產生

**選擇**: command handler 與 notification handler 共用一組格式化邏輯，但觸發來源與呼叫時機分離。

**理由**: 查詢回覆與主動通知都依賴相同資料摘要邏輯，抽離 rendering 可降低重複。

**考慮過的替代方案**:
- 每個 handler 各自組字串：容易出現格式漂移

### D6: Bot 僅回應白名單 chat_id

**選擇**: 從環境變數 `TELEGRAM_ALLOWED_CHAT_IDS` 讀取允許的 chat_id 清單（逗號分隔），只處理來自白名單的指令與訊息。

**理由**: 雖然 Bot token 本身有一定安全性，但公開部署後任何人都可能嘗試與 bot 互動。chat_id 白名單是最簡單的存取控制，不需要額外的認證流程。

**考慮過的替代方案**:
- 不加限制：任何人都能查詢帳單資料，有隱私風險
- 密碼驗證流程：增加互動複雜度，不符合 bot 的簡潔設計

### D7: `/status` 回覆中顯示 bill_id，方便 `/paid` 操作

**選擇**: `/status` 回覆的每筆帳單摘要中附上 `bill_id`，讓使用者可以直接複製用於 `/paid` 指令。

**理由**: 使用者需要知道 bill_id 才能使用 `/paid`，若不在 `/status` 中顯示，使用者無從得知 ID。

**考慮過的替代方案**:
- 使用 inline keyboard 讓使用者點選：違反 D2 純文字優先原則
- 讓使用者自行到 dashboard 查找 ID：跨平台操作不便

### D8: 多銀行帳單依銀行分組顯示

**選擇**: `/status` 與 `/summary` 的回覆中，帳單按銀行分組顯示，每個銀行區塊包含銀行名稱、帳單列表與小計。

**理由**: 使用者通常持有多家銀行信用卡，按銀行分組比混合列表更易閱讀，也方便辨識各銀行的繳費狀態。

**考慮過的替代方案**:
- 全部混合列表：帳單多時不易辨識歸屬銀行
- 每家銀行獨立指令：指令數量過多

### D9: Telegram API 呼叫採用 exponential backoff retry

**選擇**: 對 Telegram Bot API 的 sendMessage 呼叫實作 exponential backoff retry（最多 3 次，間隔 1s / 2s / 4s），遇到 rate limit（429）或暫時性錯誤（5xx）時自動重試。

**理由**: Telegram API 有速率限制，網路環境也可能出現短暫異常。指令回覆與通知推播都需要穩定送達。

**考慮過的替代方案**:
- 不重試直接失敗：使用者體驗差，通知可能靜默遺失
- 無限重試：可能在持續性問題上卡住

### D10: 通知排程由 `pipeline-scheduler` change 負責

**選擇**: 本 change 只定義通知內容格式與發送函式，實際的排程觸發（到期提醒 3 天/1 天、新帳單通知）由獨立的 `pipeline-scheduler` change 負責串接。

**理由**: 排程與通知責任分離，避免 bot 模組需要直接依賴 APScheduler 或了解排程邏輯。

## 風險 / 取捨 (Risks / Trade-offs)

**Bot 回覆過長** → 採簡潔列表格式，只顯示必要欄位
**資料尚未解析完成時無法提供摘要** → 用清楚訊息說明暫無資料或解析失敗
**通知事件來源分散** → 先定義事件輸入契約，排程觸發由 `pipeline-scheduler` change 負責
**白名單 chat_id 管理** → 從環境變數讀取，變更時需重啟服務；未來可改為資料庫管理
**非白名單使用者收到的回覆** → 靜默忽略，不回覆任何訊息以避免洩漏 bot 存在資訊
**Telegram API 暫時性失敗** → exponential backoff retry（D9）自動重試，超過次數後記錄失敗
